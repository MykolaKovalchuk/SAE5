## Migration Plan: .NET Framework 4.8 → .NET 10 (WinForms)

---

### Phase 1 — Convert All Projects to SDK-Style Format

All 17 projects use the old MSBuild format (`ToolsVersion="12.0"`). Every project needs to be rewritten as an SDK-style `.csproj`. This is the structural prerequisite for everything else.

**Target framework monikers to use:**

| Project type | `TargetFramework` | Extra properties |
|---|---|---|
| WinForms apps (`SAE5.Win`, Tools) | `net10.0-windows` | `<UseWindowsForms>true</UseWindowsForms>` |
| WinForms control libraries (`Drawing.WinForms`, `Ribbon35`) | `net10.0-windows` | `<UseWindowsForms>true</UseWindowsForms>` |
| Pure libraries with no UI (`Ravlyk.Common`, `Ravlyk.Drawing`, `Ravlyk.Drawing.ImageProcessor`, `TrueTypeSharp`) | `net10.0` or `netstandard2.0` | — |
| Libraries using `System.Drawing` (`PdfSharp`, `SAE.Drawing`, `OpenXmlPackaging`) | `net10.0-windows` | — |
| Test projects | `net10.0` | — |

### Phase 2 — Breaking Fixes (Will Not Compile)

These are hard blockers that must be resolved in order.

#### 2.1 — Remove `System.Web.Extensions` (SAE5.Win)

- Remove `<Reference Include="System.Web.Extensions" />` from SAE5.Win.csproj
- Delete the entire `<system.web>` block from App.config — it references `System.Web.ClientServices.Providers.*` which do not exist in .NET 5+. The `serviceUri` values are empty, confirming this is dead code from a VS project wizard.

#### 2.2 — Remove `System.Design` (SAE5.Win + Ribbon)

- Remove `<Reference Include="System.Design" />` from SAE5.Win.csproj
- Remove `<Reference Include="System.Design" />` from System.Windows.Forms.Ribbon35.csproj
- In the Ribbon library, locate all designer classes (e.g., `RibbonButtonDesigner`, `RibbonButtonListDesigner`) that inherit from `System.ComponentModel.Design.*` and either delete them or stub them with empty classes. They are only used in the VS designer — runtime behavior is unaffected.

#### 2.3 — Fix `WindowsBase` in OpenXmlPackaging

`System.IO.Packaging` was in `WindowsBase.dll` (a WPF assembly) on .NET Framework but is a standalone NuGet on .NET Core+.

- Remove `<Reference Include="WindowsBase" />` from Ravlyk.Adopted.OpenXmlPackaging.csproj
- Add `<PackageReference Include="System.IO.Packaging" Version="8.0.0" />` — the API (`Package.Open`, `PackagePart`, etc.) is identical.

#### 2.4 — Remove `System.Deployment` (Tools)

- Remove `<Reference Include="System.Deployment" />` from ImageConverter.csproj and PaletteFileEditor.csproj. ClickOnce/deployment APIs do not exist in .NET 5+. Since these are utility tools, ClickOnce is almost certainly not used.

### Phase 3 — Runtime Failures (Compile OK, Break at Runtime)

#### 3.1 — Add `System.Drawing.Common` NuGet

On .NET 6+, `System.Drawing.Common` is Windows-only and must be explicitly referenced. Add `<PackageReference Include="System.Drawing.Common" Version="9.x" />` to every project that uses `System.Drawing.*`:

- SAE5.Win.csproj
- Ravlyk.Adopted.PdfSharp.csproj — heavily uses `System.Drawing.Font`, `Graphics`, `Bitmap`, `PrivateFontCollection`, `Metafile`
- Ravlyk.SAE.Drawing.csproj
- Ravlyk.Drawing.WinForms.csproj
- Ravlyk.Adopted.OpenXmlPackaging.csproj
- System.Windows.Forms.Ribbon35.csproj
- ImageConverter.csproj
- PaletteFileEditor.csproj

#### 3.2 — Fix `Process.Start(url)` in 6 files

On .NET Core+, `UseShellExecute` defaults to `false`. Any call that opens a URL or file with `Process.Start(string)` will throw a `Win32Exception` at runtime.

Replace all occurrences with:
```csharp
Process.Start(new ProcessStartInfo("https://...") { UseShellExecute = true });
```

Affected locations:
- AppInfo.cs (2 locations)
- RegisterDialog.cs
- UpdateDialog.cs
- HomeUserControl.cs (2 locations)
- SchemeUserControl.cs
- PrintUserControl.cs

#### 3.3 — Add `System.Configuration.ConfigurationManager` NuGet

`System.Configuration` is not in the .NET 10 BCL. The app uses `ApplicationSettingsBase` / `Settings.Default` (Program.cs).

- Add `<PackageReference Include="System.Configuration.ConfigurationManager" Version="9.x" />` to SAE5.Win.csproj
- The `Properties.Settings` mechanism continues to work with this package.
- Clean up App.config: remove the `<startup>` / `<supportedRuntime>` element — .NET 10 ignores it.

#### 3.4 — Replace `WebClient` with `HttpClient`

`System.Net.WebClient` is `[Obsolete]` since .NET 5 (AppInfo.cs):
```csharp
// Before
using (var webClient = new WebClient())
    return webClient.DownloadString(fileUrl);

// After — inject or use a static HttpClient
using var http = new HttpClient();
return await http.GetStringAsync(fileUrl);
```
This requires making the surrounding `CheckForUpdatesCore` method `async`.

### Phase 4 — Dependency Library Specific Notes

#### PdfSharp (vendored, ~100 source files)
- **Status**: No unsafe code, no P/Invokes, only `System.Drawing` dependency.
- **Action**: After adding `System.Drawing.Common` (Phase 3.1) and converting to SDK-style (Phase 1), this library should compile unchanged.
- **Optional upgrade**: Replace the entire vendored copy with the maintained [PdfSharp 6.x NuGet package](https://www.nuget.org/packages/PdfSharp/) which natively targets .NET 6+. This would require updating the API calls in Ravlyk.SAE.Drawing that use the `Ravlyk.Adopted.PdfSharp` namespace, but would eliminate maintenance burden going forward.

#### TrueTypeSharp (vendored)
- **Status**: Zero external dependencies — pure managed C# with no P/Invoke, no unsafe code, no platform-specific logic. Uses `FakePtr<T>` for C-style pointer simulation.
- **Action**: Change target to `net10.0` (or `netstandard2.0`). **No code changes needed.**

#### OpenXmlPackaging (vendored)
- **Status**: One breaking issue (`WindowsBase`) fixed in Phase 2.3.
- **Action**: Swap `WindowsBase` for the `System.IO.Packaging` NuGet. Done. No code changes needed.

#### System.Windows.Forms.Ribbon (vendored, vintage 2009)
- **Status**: Most complex adopted library. Uses `System.Design` for designer support (breaking), 20+ Win32 P/Invokes (fine on Windows), custom `System.Windows.Forms` namespace (may cause ambiguity).
- **Actions**:
  1. Remove designer classes (files with names like `*Designer.cs`) or convert them to empty stubs — they are only consumed by the VS Forms Designer, not at runtime.
  2. Remove `System.Design` reference.
  3. Add `System.Drawing.Common` NuGet.
  4. Add a null-check guard on `Process.GetCurrentProcess().MainModule` in GlobalHook.cs — `MainModule` can return null on .NET in certain conditions.
